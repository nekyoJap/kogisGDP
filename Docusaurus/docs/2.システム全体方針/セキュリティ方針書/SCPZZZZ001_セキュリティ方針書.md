## 本ドキュメントの位置づけ

本書は、セキュリティに関する攻撃からどう守り、データをどう保護するか（Defense & Protection）を定義する。

特に、**ID管理以外の防御壁（Guardrail）** に関する方針も本書の対象とする。

本書の内容は、以下の横断アーキテクチャ・非機能設計文書に拘束力を持つ：

**表1 — 本書の拘束対象**

| 対象文書 | 本書との関係 |
|---------|------------|
| セキュリティ非機能要件定義書 | 技術レベル・暗号強度・応答時間・閾値を具体化 |
| ロギング／監査方針書 | 記録項目・保管期間・アクセス制御の設計 |
| API仕様書／画面仕様書／外部連携仕様書 | 本書で定めたセキュリティ原則に準拠 |
| システムIaC構成定義書／運用要件定義書 | インフラ・コンテナ・鍵管理などの実装詳細 |

**テーマ：**

> 「攻撃からどう守り、データをどう保護するか」（Defense & Protection）  
> ここには AuthN/AuthZ 以外の **防御壁（Guardrail）** が集約される。

## 第3章 用語及び定義

本書で使用する主要な用語を以下に定義する。

**表5 — 用語定義**

| 用語 | 定義 |
|------|------|
| PII | Personally Identifiable Information（個人識別情報） |
| At Rest | 保存時（ストレージ上のデータ） |
| In Transit | 転送時（ネットワーク通信中のデータ） |
| トークナイズ | 機密データを非機密のトークンに置き換える処理 |

## 第4章 セキュリティ基本原則（多層防御観点）

### 4.1 セキュリティ目標

本システムは以下のセキュリティ目標を達成しなければならない：

**表6 — セキュリティ目標**

| 目標 | 説明 |
|------|------|
| 機密性（Confidentiality） | 許可された者のみが情報にアクセスできる状態を維持する |
| 完全性（Integrity） | 情報および処理方法が正確かつ完全である状態を維持する |
| 可用性（Availability） | 許可された者が必要な時に情報およびシステムにアクセスできる状態を維持する |
| 責任追跡性（Accountability） | すべての操作を追跡し、責任を明確にできる状態を維持する |
| 防御継続性（Resilience） | 攻撃や障害に対して迅速に回復できる能力を維持する |

### 4.2 防御設計の原則（Defense-in-Depth）

多層防御の原則に基づき、以下の方針を採用しなければならない。

**表7 — 防御設計の原則**

| 観点 | 方針 |
|------|------|
| 多層防御 | 複数の防御層（認証・API・WAF・通信・DB・監査）で保護しなければならない |
| ゼロトラスト | 信頼境界を設けず、全ての通信・操作を検証しなければならない |
| 最小権限 | すべてのアクセスは必要最小限としなければならない |
| 安全なデフォルト | Default Deny / Private を原則としなければならない |
| 侵害前提設計 | 「侵入される」ことを前提とし、検知・封じ込めを重視しなければならない |
| セキュアバイデザイン | 設計段階からセキュリティを組み込まなければならない |

## 第5章 認証・認可との関係整理（境界定義）

認証・認可方式そのものは「認証・認可方針書」に記載する。

本書では、防御設計に関わる以下の内容のみ記載する。

**表8 — 本書で扱う認証・認可関連項目**

| 項目 | 本書で扱う観点 |
|------|---------------|
| トークン保護 | 暗号化、漏洩時の失効、保管禁止、ログ禁止 |
| セッション保護 | 固定化・盗聴防止／Secure Cookie／HttpOnly |
| API権限制御 | レートリミット／API Gateway／WAF検知ブロック |
| 認証情報保護 | パスワードハッシュ化、ソルト付与、保管方式 |

**注記：** RBAC / ABAC / MFA / OAuth2.0 の方式定義は扱わない（認証・認可方針書で管理）

## 第6章 データ保護・プライバシー方針

### 6.1 データ分類

機密性に応じてデータを分類し、適切な保護レベルを適用しなければならない。

**表9 — データ分類**

| 区分 | 例 | 保護レベル | 保護要件 |
|------|-----|-----------|----------|
| 特定個人情報 | マイナンバー等 | 最高 | 暗号化＋トークナイズ＋アクセス制限＋監査 |
| 決済機密情報 | カード番号／CVV／暗証番号 | 最高 | PCI DSS準拠、暗号化、マスキング |
| 個人情報 | 氏名／住所／連絡先／生年月日 | 高 | 暗号化＋マスキング＋アクセス制限 |
| 与信審査情報 | 属性／スコア／申込履歴 | 高 | 暗号化＋アクセス制限＋監査 |
| 業務情報 | マスタ／取引情報 | 中 | アクセス制限＋ログ記録 |
| 公開情報 | 店舗一覧等 | 低 | 整合性検証 |

### 6.2 暗号化・マスキング・トークナイズ方針

**表10 — 暗号化・マスキング・トークナイズ方針**

| 項目 | 方針 |
|------|------|
| 保存時暗号化（At Rest） | DB / S3 / Backup に対して AES-256 以上で暗号化を実施しなければならない |
| 通信時暗号化（In Transit） | すべての通信経路を TLS 1.3 以上に統一しなければならない |
| マスキング | 画面・ログ・監査対象では機微情報を適切にマスキングしなければならない |
| トークナイズ | PII・決済情報にはトークナイズを優先採用することが望ましい |
| ハッシュ化 | パスワードなど不可逆データは SHA-256 以上のハッシュで保存しなければならない |
| ソルト | パスワードハッシュにはソルトを付与しなければならない |
| 鍵管理 | 暗号鍵は専用の鍵管理サービス（KMS等）で管理しなければならない |
| ログ制御 | 生データ、トークン、鍵情報、パスワードをログに記録してはならない |

### 6.3 PII保護の特別要件

個人情報保護法および金融業界の規制に準拠し、以下を実施しなければならない。

**表11 — PII保護要件**

| 要件 | 内容 |
|------|------|
| 収集最小化 | 必要最小限のPIIのみを収集しなければならない |
| 利用目的明示 | PII収集時に利用目的を明示しなければならない |
| 同意取得 | PII利用前に本人の同意を取得しなければならない |
| アクセス制御 | PIIへのアクセスは役割ベースで制限しなければならない |
| 保管期間制限 | 法定保管期間経過後は速やかに削除しなければならない |
| 本人開示 | 本人からの開示請求に対応できる仕組みを整備しなければならない |
| 削除要求対応 | 本人からの削除要求に対応できる仕組みを整備しなければならない |

## 第7章 インタフェース別セキュリティ方針

### 7.1 API

**表12 — APIセキュリティ方針**

| 観点 | 方針 |
|------|------|
| WAF | SQL Injection / XSS 試行検知で即時遮断しなければならない |
| レート制限 | 外部API／公開APIにはレート制限を適用しなければならない |
| 入力検証 | 想定外／未定義形式のリクエストは拒否しなければならない |
| 出力エンコーディング | レスポンスは適切にエンコーディングしなければならない |
| CORS設定 | 必要最小限のオリジンのみを許可しなければならない |
| APIキー保護 | APIキーは環境変数または秘密管理サービスで管理しなければならない |
| 改ざん検知 | 重要APIには署名付きリクエストを採用することが望ましい |
| バージョニング | セキュリティ問題のあるAPIバージョンは速やかに廃止しなければならない |

### 7.2 画面（UI）

**表13 — 画面セキュリティ方針**

| 観点 | 方針 |
|------|------|
| CSRF対策 | トークン制御（Synchronizer Token Pattern等）を実施しなければならない |
| XSS対策 | 出力エスケープ＋CSP（Content Security Policy）設定を実施しなければならない |
| セッション管理 | Secure / HttpOnly / SameSite 属性を適切に設定しなければならない |
| クリックジャッキング対策 | X-Frame-Options または CSP の frame-ancestors を設定しなければならない |
| パスワード入力 | オートコンプリート無効化を検討しなければならない |
| 機微情報表示 | 必要に応じてマスキング表示を実施しなければならない |
| スクリーンショット対策 | 機微情報画面ではマスキング等を実装することが望ましい |
| セッションタイムアウト | 一定時間操作がない場合は自動ログアウトしなければならない |

### 7.3 外部連携

**表14 — 外部連携セキュリティ方針**

| 観点 | 方針 |
|------|------|
| 接続先検証 | TLS証明書検証を実施しなければならない |
| 相互認証 | 重要な外部連携には相互TLS認証を採用することが望ましい |
| データ最小化 | 最小限のデータのみを送信しなければならない |
| タイムアウト設定 | 適切なタイムアウト値を設定しなければならない |
| リトライ制御 | 無限リトライを避け、上限を設定しなければならない |
| 依存サービスの脆弱性 | 定期スキャン・更新ルールを策定しなければならない |
| エラーハンドリング | 詳細なエラー情報を外部に公開してはならない |
| ホワイトリスト管理 | 接続可能な外部システムをホワイトリスト管理しなければならない |

## 第8章 開発／運用における防御方針（DevSecOps）

### 8.1 Secure Coding

開発者は以下のセキュアコーディング指針に従わなければならない。

**表15 — Secure Coding指針**

| 項目 | 指針 |
|------|------|
| SQL Injection対策 | ORM使用／プリペアドステートメント／文字列結合を禁止しなければならない |
| XSS対策 | 出力エスケープ／CSP設定を実施しなければならない |
| CSRF対策 | Token方式を採用しなければならない |
| パストラバーサル対策 | ファイルパスのサニタイゼーションを実施しなければならない |
| コマンドインジェクション対策 | 外部コマンド実行は最小化し、入力を検証しなければならない |
| ログ | 生パスワード／機密情報の出力を禁止しなければならない |
| 入力検証 | 型、長さ、形式、範囲、ホワイトリストによる制御を実施しなければならない |
| エラーメッセージ | スタックトレース等の詳細情報を本番環境で公開してはならない |
| ハードコード禁止 | 認証情報・APIキー・秘密鍵をコードに埋め込んではならない |

### 8.2 サプライチェーン・依存関係管理

**表16 — サプライチェーンセキュリティ方針**

| 項目 | 方針 |
|------|------|
| SBOM管理 | 使用OSS／依存関係は全て可視化しなければならない |
| 脆弱性スキャン | ライブラリ更新時・リリース前に脆弱性スキャンを実施しなければならない |
| 依存関係の最小化 | 不要な依存関係は削除しなければならない |
| ライセンス確認 | OSSライセンスの互換性を確認しなければならない |
| 信頼できるソース | 公式リポジトリからのみ依存関係を取得しなければならない |
| バージョン固定 | 依存関係のバージョンを固定し、意図しない更新を防止しなければならない |
| 定期更新 | セキュリティパッチは速やかに適用しなければならない |

### 8.3 AI開発防御・コード検閲

**表17 — AI開発防御方針**

| 項目 | 方針 |
|------|------|
| AI Linter | セキュリティ禁則事項を自動チェックすることが望ましい |
| 秘密情報 | Git／ログ／AIチャット上で秘密情報を共有してはならない |
| コードレビュー | セキュリティ観点でのコードレビューを実施しなければならない |
| 静的解析 | SAST（Static Application Security Testing）ツールを導入することが望ましい |
| 動的解析 | DAST（Dynamic Application Security Testing）ツールを導入することが望ましい |

## 第10章 ログ・監査・証跡に関する方針

セキュリティログ・監査証跡の詳細は「アプリケーションログ方針書」を参照すること。

本章では、セキュリティ観点での要求事項のみを記載する。

**表21 — ログ・監査・証跡方針**

| 観点 | 方針 |
|------|------|
| 記録対象 | 認証／認可／重要データアクセス／設定変更／権限変更を記録しなければならない |
| ログ保護 | WORM／署名／アクセス制限を適用し、改ざんを防止しなければならない |
| トレーサビリティ | 誰が／いつ／何にアクセスしたか記録しなければならない |
| 不正検知 | 合理的な閾値・異常検知ルールを定義しなければならない |
| 保管期間 | 法的要件に基づいた保管期間を設定しなければならない |
| リアルタイム監視 | 重大なセキュリティイベントはリアルタイムで検知しなければならない |
| アラート | 不正アクセスや異常な操作を検知した際はアラートを発報しなければならない |

### 11.1 インシデント対応体制

セキュリティインシデント発生時の対応体制を整備しなければならない。

**表22 — インシデント対応体制**

| フェーズ | 対応内容 |
|---------|---------|
| 検知 | セキュリティイベントの監視・異常検知 |
| 分析 | インシデントの影響範囲・重大度の評価 |
| 封じ込め | 被害拡大の防止、攻撃元の遮断 |
| 根絶 | 脆弱性の修正、マルウェアの除去 |
| 復旧 | システムの正常稼働への復帰 |
| 事後対応 | 原因分析、再発防止策の策定、報告 |

### 11.2 インシデント分類

**表23 — インシデント分類と対応レベル**

| レベル | 説明 | 例 | 対応時間 |
|--------|------|-----|---------|
| Critical | システム全体に影響 | 大規模データ漏洩、システム停止 | 即座 |
| High | 重要機能に影響 | 特定機能の脆弱性悪用 | 1時間以内 |
| Medium | 部分的影響 | 軽微な脆弱性検知 | 4時間以内 |
| Low | 影響軽微 | ポリシー違反の検知 | 24時間以内 |

### 11.3 情報開示・報告

**表24 — 情報開示・報告方針**

| 項目 | 方針 |
|------|------|
| 内部報告 | セキュリティインシデントは速やかに経営層に報告しなければならない |
| 外部報告 | 法的要件に基づき、監督官庁への報告を実施しなければならない |
| 顧客通知 | 個人情報漏洩時は影響を受ける顧客に通知しなければならない |
| 公表 | 重大インシデントは適切なタイミングで公表を検討しなければならない |

## 付録B 検討が必要な論点

以下は、本方針書の運用にあたり検討が必要な論点である。

1. **PCI DSS準拠の詳細要件**
   - カード情報処理における具体的な技術要件の確認
   - PCI DSS認証取得の要否とスケジュール
   - QSA（認定セキュリティ査定機関）による監査計画

2. **トークナイゼーションサービスの選定**
   - カード番号・口座情報のトークナイゼーション実装方式
   - サードパーティサービス利用 vs 自社実装の判断
   - トークン管理基盤の構築

3. **ゼロトラストアーキテクチャの実装範囲**
   - BC間通信における認証・認可の実装レベル
   - マイクロサービス間の相互認証方式
   - サービスメッシュ導入の検討

4. **脆弱性管理プロセスの整備**
   - 脆弱性スキャンツールの選定
   - 発見された脆弱性の優先度付けルール
   - パッチ適用の承認プロセス

5. **セキュリティ監視体制の構築**
   - SIEM（Security Information and Event Management）導入
   - SOC（Security Operations Center）体制の検討
   - 24時間監視の要否と体制

6. **データ保管期間とプライバシー規制**
   - 個人情報保護法、GDPR等の規制対応
   - データ保管期間と削除ルールの策定
   - 本人からのデータ削除要求への対応フロー

7. **クラウドセキュリティ設定基準**
   - AWS、Azure等のセキュリティベストプラクティス適用
   - IaC（Infrastructure as Code）によるセキュリティ設定の統制
   - クラウド設定の継続的監査

8. **サプライチェーンリスク管理**
   - ベンダーセキュリティ評価基準
   - 委託先のセキュリティ管理
   - ソフトウェアサプライチェーン攻撃への対策
